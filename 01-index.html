SGBDR ou SGBD

  • SGBDR : Systéme de gestion de base de données relationnelle
  • SGBD : Systéme de gestion de base de données
  • MCD : Modéle conceptuel de données
  • BDD : Base de donnees
  • DB : Database (anglais)
  • SQL : Structured Query langage (langage de requêtes structurées)

En résumé

  • MySQL est un SGBDR, un systeme de gestion de base de données relationnelles
  • Le Langage SQL est utilisé pour la communication entre le client et le serveur
  • Dans une BDD les données sont représentées sous forme de tables
  • Dans une table, nous avons des colonnes, chaque nouvelle donnée sera introduite sous forme de ligne dans la table

1- Modélisation d'un BDD

Avant de créer notre base de données, il est essentiel de se poser et de réfléchir à la modélisation. Rien est irréversible, mais changer la modélisation en cours de projet est deconseillée car cela forcera certainement à réecrire une partie du code ce qui peut prendre beaucoup de temps et avoir des conséquence sur ce qu'on a déja fait comme des déréglement d'autres parties de l'application. C'est la raison pour laquelle il vaut éviter de se tromper

Comment modéliser une BDD

Si la BDD n'est pas complexe, il est possible de la modéliser directement avec une feuille et un stylo. Cependant dans la plupart des cas nous aurons besoin d'un logiciel pour accompagner et avoir une vue d'ensemble Un logicile connu permet de modéliser une BDD, c'est MySQL Morklecht. Il existe cependant de nombreux outils permettant cette manipulation. Pour une bonne conception de sa base de données il faut réfléchir en terme de sujet en modélisant ceux du monde réel. Par exemple si nous vendons des produits sur notre site il y aura une table utilisateur, une table produits et une table commande (cette table rassemblera id de l'utilisateur et celui du produit)

2- Les tables

  • Un sujet représente un table dans une BDD
  • Une table est un emplacement de sauvegarde
  • Les donnée sont stockées à l'intérieur de tables. Une tables peut être comparée à une liste qui contient des enregistrement relatifs à un sujet bien défini

Il faut en fait, dés le départ, réfléchir à chaque sujet abordé dans notre site et à ses conséquences. Si nous vendns des produits, nous aurons certainement des membres Une table menbre (user) sera donc nécessaire. En terme de fonctionnalités. Si nous souhaitons proposer un abonnnement Newsletter. nous aurons besoin d'une table newsletter qui sera liée à la table menbre (user), table = 1 sujet

3- Les colonnes ou les champs

Chaque table posséde generalement plusieur champs , aussi appelé colonne. Les colonnes présentent des caractéristiques relatives au sujet. (-> table)

Pour savoir quel colonne mettre dans quelle table, il faut se poser les question suivante : qu'est ce qui pourrait décrire mon sujet ? quelles sont les informations sur mon sujet ?
La table user aura par exemple les champs : pseudo, mot de passe, nom, prenom, etc...

5- Types de données, PK, AI pour les identifiants

Les identifiants, PK (Primary Key - Clé primaire)

  • Les identifiants sont des champs un peu particulier car c'est la seule colonne qui ne décrit pas le sujet. Ils sont cependant systématiquement présent dans chaque table en premere position
  • Chaque table possède une colonne identifiant que l'on appelera la clé primaire ou tout simplement PK
  • On pourra l'appeler tout simplement 'id' mais on retrouvera parfois, 'idProduit', ou encore 'idMembre' etc..

A, l'auto incrémentation

Auto Increment est une option permettant de générer un numéro unique dans une colonne de type clé primaire (PK). Par conséquent, retenez bien que le premier champ de chaque table sera systématiquement un 'id' qui sera à la fois PK (Primary Key) et AI (Auto_Increment)

NULL / NOT NULL

Dans chaque champs, nous pourrons indiquer si nous acceptons les valeur NULL ou NOT NULL. NULL est un type de valeur en informatique permettant de laissez un champs vide si nous n'avons pas d'informations à y laisser

Les relations

Il arrive parfois que les sujets interagissent entre eux

  • Exemple : un 'membre' commande un 'produit', nous devons enregistrer l'informations dans une table 'commande'
  • Autre exemple : Le conducteur conduit un véhicule. Il faudra une table permettant de préciser quel conducteur va avec quel véhicule

6- Les cardinalités

Les cardinalités permettent de connaître le chiffre maximun et minimun d'enregistrement pour une relation.

Par exemple, un enfant est la descendance d'une seule mére biologique (minimun = 1 / maximun = 1). En revanche, une femme peu avoir plusieurs enfant ou aucun (minimum = 0 / maximum = n).

Autre exemple, 1 livre possède forcément 1 auteur et 1 max (1,1).

7- Table de jointures

Une table de jointure permet de faire le lien entre deux tables. Par exemple prenoms le cas d'une société de taxis qui posséderait des conducteurs (table conducteurs) et des véhicules (tables véhicules)

  • Un conducteur peut conduire 1 ou plusieurs véhicules
  • Un véhicule peut être conduit par 0 ou plusieurs conducteur

Comment savoir quel conducteur conduit quel véhicule

Grâce à la table de jonture, nous savons qu'Alexandre conduit à la fois la Mercedes et la BMW et que Julien que la BMW

Les clés étrangére, Foreign Key (FK)

Les clés étrangere sont des champs forcement clé primaire dans leur table d'origine et qui sont présent dans une table extérieure.
Un champ clé étrangere (FK) placé à l'extéreur de sa table d'origine permet de la relation avec un autre sujet

9- Les enregistrement

Chaque enegistrement représente une ligne dans la BDD et chaque enregistrement est auto incrémenté. Même si un enregistrement est supprimer sont id ne sera pas réutilisé

10- Comment interrogé la BDD ?

Les requêtes

Une requête permet de poser une question, d'interroger la BDD, ou encore de donner un ordre (crudo -> Create ->créer / Read -> lis une info / update -> met à jour / Delete -> supprime)

Nous terminerons toutes nos requêtes SQL par un point virgule pour signifier qu'il faut éxécuter la demande

Types de requête

Il existe 4 types de requêtes

  • La sélection : SELECT - c'est une question / réponse => nous faisons une demande et obtenons une réponse
  • L'insertion : INSERT : requête d'action -> impact sur les données
  • La Modification : UPDATE - requête d'action -> on modifie les données déjà présentes dans notre table
  • La suppression : DELETE - requête d'action -> on supprime un enregistrement de notre table // une table // une BDD
